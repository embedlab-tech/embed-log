# Deterministic UI tests plan

This plan defines the UI regression tests we want to support once the demo traffic generator has a deterministic test profile.

Related docs:

- `tests-ui/TEST_PLAN.md` — broad UI/E2E regression plan
- `tests-ui/DETERMINISTIC_DEMO_PLAN.md` — deterministic demo profile implementation plan

## Goal

Move UI tests away from random demo traffic and toward predictable test data.

Tests should be able to wait for known payloads such as:

```text
TEST src=SENSOR_A tick=020 seq=0061 kind=sync msg="controller state stable"
```

instead of relying on random log counts or arbitrary sleeps.

## Time strategy

We will not mock wall-clock time initially.

Assumption:

- absolute timestamps generated by the backend can vary per run
- deterministic payload `tick=N` and `seq=N` are the stable identifiers
- tests assert ordering, range inclusion, and approximate sync behavior using ticks

## Required deterministic demo properties

The test demo profile should guarantee:

1. all configured panes receive logs quickly
2. each source emits stable `TEST src=... tick=... seq=... kind=...` lines
3. related source events share the same tick number
4. same-tick events are emitted close together in time
5. special cases appear at predictable ticks:
   - duplicated source prefix
   - embedded timestamp prefix
   - warning/error tags
   - filterable messages
6. traffic continues until the test run ends

## Proposed deterministic message kinds

```text
kind=sync              normal synchronized cross-pane event
kind=warning           stable warning line
kind=error             stable error line
kind=prefix-cleanup    message starts with [SENSOR_X]
kind=timestamp-cleanup message starts with [2026-...]
kind=filter-alpha      known text for filter tests
kind=filter-beta       known text for negative/alternate filter tests
```

Example schedule:

```text
every tick       kind=sync for SENSOR_A/B/C
tick % 5 == 0   kind=warning
tick % 7 == 0   kind=error
tick % 9 == 0   kind=prefix-cleanup
tick % 11 == 0  kind=timestamp-cleanup
tick % 13 == 0  kind=filter-alpha
tick % 17 == 0  kind=filter-beta
```

- `#ws-status` contains `connected`
- panes `SENSOR_A`, `SENSOR_B`, `SENSOR_C` exist

- prove deterministic profile is running
- protect WebSocket/config/render path

# Console/error policy

All tests should collect and fail on unexpected page errors.

Recommended helper:

```js
export function collectPageErrors(page) {
  const errors = [];
  page.on('pageerror', err => errors.push(String(err)));
  page.on('console', msg => {
    if (msg.type() === 'error') errors.push(msg.text());
  });
  return errors;
}
```

At the end of each test or in `afterEach`:

```js
expect(errors).toEqual([]);
```

Allowlist only if absolutely necessary.

## Acceptance criteria for deterministic test suite

- no random sleeps except short browser stability waits if unavoidable
- tests wait for known text/ticks
- tests pass repeatedly on local macOS
- test logs are isolated under `tests-ui/.tmp/logs`
- failed tests retain Playwright trace/screenshot/video
- deterministic profile is fast enough for routine local runs
